# Redrob Candidate Ranker

A multi-stage AI candidate ranking pipeline for the Redrob Intelligent Candidate Discovery & Ranking Challenge.

**Input:** 100,000 candidate profiles (JSONL) + job description  
**Output:** `submission.csv` — top 100 candidates, ranked, with scores and reasoning  
**Constraints:** ≤5 min wall-clock, ≤16 GB RAM, CPU-only, no network during `rank.py`

---

## Architecture

```
PRE-COMPUTATION (no time limit — run once)
──────────────────────────────────────────
candidates.jsonl.gz
    ├── scripts/01_extract_features.py  ──► precomputed/features.parquet
    ├── scripts/02_embed_candidates.py  ──► precomputed/candidate_embeddings.npy
    │                                       precomputed/candidate_ids.txt
    │                                       precomputed/candidate_texts.pkl
    └── scripts/03_embed_jd.py          ──► precomputed/jd_embedding.npy

RANKING STEP (≤5 min, CPU, no network)
───────────────────────────────────────
Stage 1: BM25 keyword retrieval           pipeline/bm25_retrieval.py
Stage 2: Dense cosine similarity          pipeline/dense_retrieval.py
Fusion:  weighted score fusion → top-5000 pipeline/fusion.py
Stage 3: Structural + availability        scoring/
Stage 4: Cross-encoder reranker → top-100 pipeline/cross_encoder.py
Output:  submission.csv  (candidate_id, rank, score, reasoning)
```

## Execution Order

```bash
# Step 1: Feature extraction (~5 min for 100K candidates)
python scripts/01_extract_features.py

# Step 2: Embed candidates (~3-4 hours on CPU — run overnight)
python scripts/02_embed_candidates.py

# Step 3: Embed JD query (fast — seconds)
python scripts/03_embed_jd.py

# Step 4: Run ranking
python rank.py --out submission.csv

# Optional: offline ablation eval (requires eval/manual_labels.json)
python scripts/04_eval.py

# Optional: validate submission format
python dataset/validate_submission.py submission.csv

# Optional: run sandbox demo
streamlit run sandbox/app.py
```

### Useful rank.py flags

| Flag | Default | Purpose |
|------|---------|---------|
| `--precomputed` | `precomputed/` | Path to precomputed files |
| `--out` | `submission.csv` | Output CSV path |
| `--ce-topn` | `500` | Candidates entering cross-encoder |
| `--fusion-topn` | `5000` | Shortlist size after fusion |
| `--no-reasoning` | off | Skip reasoning (faster, outputs IDs only) |
| `--debug` | off | Write per-stage CSVs to `bin/` — dev only, not for submission |

## Precomputed Files (Git LFS)

Large precomputed files are tracked via **Git LFS**. They do not need to be regenerated unless you change the feature extraction or embedding logic.

| File | Size | Generated by |
|------|------|-------------|
| `precomputed/candidate_embeddings.npy` | ~154 MB | `scripts/02_embed_candidates.py` |
| `precomputed/candidate_texts.pkl` | ~50 MB | `scripts/02_embed_candidates.py` |
| `precomputed/features.parquet` | ~15 MB | `scripts/01_extract_features.py` |
| `precomputed/candidate_ids.txt` | <1 MB | `scripts/02_embed_candidates.py` |
| `precomputed/jd_embedding.npy` | <1 MB | `scripts/03_embed_jd.py` |

### First-time clone (with LFS files)

```bash
# Requires Git LFS installed: https://git-lfs.github.com
git lfs install
git clone https://github.com/Aayushman-nvm/recruiter-ai
```

If you cloned without LFS and got pointer files instead of real data:

```bash
git lfs pull
```

### Setting up LFS in a fresh repo

```bash
git lfs install
git lfs track "*.npy" "*.parquet" "*.pkl"
git add .gitattributes
git commit -m "track large files with LFS"
git push
```

The `.gitattributes` in this repo already has these rules — `git lfs install` is all you need before cloning.

### Regenerate from scratch

If you want to verify the full pipeline without using the precomputed files:

```bash
python scripts/01_extract_features.py
python scripts/02_embed_candidates.py
python scripts/03_embed_jd.py
python rank.py --out submission.csv
```

## Offline Evaluation

Run on `dataset/sample_candidates.json` (50 candidates) with manual relevance labels.

First, create `eval/manual_labels.json`:
```json
{ "CAND_0000001": 3, "CAND_0000002": 1, ... }
```
Relevance scale: 0 = not relevant, 1 = marginal, 2 = relevant, 3 = highly relevant.

Then run:
```bash
python scripts/04_eval.py
```

Results are saved to `eval/eval_results.json`.

## Key Design Decisions

- **Encoder:** `BAAI/bge-small-en-v1.5` (384-dim). bge-small hits a good quality/speed balance on constrained CPU hardware (Ryzen 3250U: ~18h vs ~72h for bge-large).
- **Cross-encoder:** `cross-encoder/ms-marco-MiniLM-L-6-v2` on top-500 by prelim score. ~35s on CPU.
- **BM25 hybrid:** Catches exact technical terms (FAISS, Pinecone, NDCG) that dense vectors smooth over. Weighted score fusion (not RRF) preserves confidence levels — a dense score of 0.95 vs 0.32 is lost in RRF.
- **jd_query.txt:** ~170-word focused requirements query for embedding. Full `jd.txt` is used only for BM25 tokens.
- **Structural scoring:** Five hard-disqualifier rules (IT-services-only career, research-only career, shallow recent ML, CV/speech-without-NLP, disqualifying language) all hard-zero or multiply-penalise scores. Ghost-skill detection catches keyword stuffing.
- **Weight config:** All blend weights live in `config/weights.py` — single source of truth for `rank.py`, `sandbox/app.py`, and `scripts/04_eval.py`.
- **Availability:** 0.12 prelim weight — tiebreaker signal, not a primary driver. Five sub-signals: open_to_work, recency, response_rate, notice_period, response_time.

## Sandbox Demo

The demo at `sandbox/app.py` uses the same `pipeline/feature_extraction.py`, scoring functions, and `generate_reasoning()` as `rank.py` — no formula drift.

Upload a JSON array of ≤100 candidate objects, and it will:
1. Embed with `BAAI/bge-small-en-v1.5`
2. Score with the full structural + availability pipeline
3. Rerank with the cross-encoder
4. Show a score component breakdown chart, top-10 reasoning, and a downloadable CSV
